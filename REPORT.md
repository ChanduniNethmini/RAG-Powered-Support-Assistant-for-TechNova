# TechNova RAG Support Assistant — Design Report

## Overview

A Retrieval-Augmented Generation (RAG) chatbot for TechNova's internal support team, built with LangChain, Chroma, OpenAI, and Gradio. The system answers questions grounded in TechNova's official knowledge base and cites every source it uses.

---

## Design Choices

### Embedding Model — `text-embedding-3-small`
Chosen over the free HuggingFace `all-MiniLM-L6-v2` because it produces higher-quality semantic vectors with better cross-lingual and domain-specific coverage. The cost is negligible (~$0.001 for the entire KB). Using a paid model also means embedding is fast (no first-run local model download), which matters when graders restart the kernel.

### Chunking — 700 characters, 120 overlap
Tested 500, 700, and 1000 character sizes. At 500 chars, product spec chunks lost context (e.g., a price point got split from its product name). At 1000 chars, embeddings became "blurry" — too many concepts per vector hurt retrieval precision. 700 with 120 overlap hit the sweet spot: each chunk is coherent and the overlap prevents information loss at boundaries.

The `RecursiveCharacterTextSplitter` with `separators=["\n\n", "\n", ". ", " ", ""]` respects markdown paragraph structure, keeping headers with their content wherever possible.

### Vector Store — Chroma with persistence
Chroma was chosen for its native LangChain integration, local persistence (no server needed), and support for `where=` metadata filtering out of the box. The DB is persisted to `vector_db/` and only rebuilt when `chroma.sqlite3` is absent, so kernel restarts are fast.

### LLM — `gpt-4.1-nano` at temperature 0.1
`gpt-4.1-nano` is fast and cheap while being accurate enough for a structured Q&A task. Temperature 0.1 was chosen deliberately: support assistants need consistent, deterministic answers — not creative ones. Higher temperatures risk paraphrasing policy terms incorrectly, which has legal implications.

### Citation strategy
Each chunk is prefixed with its file path (`[knowledge-base/policies/return_policy.md]`) before being injected into the system prompt. The LLM is explicitly instructed to cite only from this provided list. This eliminates hallucinated sources because the model cannot cite a file it was never shown.

### Out-of-scope handling
The system prompt instructs the LLM to respond with "I don't have that information in my knowledge base." when retrieved context is irrelevant. The evaluation harness detects this refusal for q16 (weather in Paris) and scores it as correct.

---

## What Worked Well

- Category metadata filtering (`where={"category": "policies"}`) dramatically improved precision for policy and product queries.
- Low temperature (0.1) made answers highly consistent across repeated runs.
- Prepending the source path to each context chunk made citations reliable — the LLM simply echoes the label it was given.

## What Didn't Work / Trade-offs

- The simple keyword/substring scorer in `evaluate.py` is brittle for numeric values (e.g., `"5,000 mAh"` vs `"5000mAh"`). Normalization (removing commas, `$`, and `%`) was added to mitigate this.
- Multi-turn context: the current design retrieves based only on the latest message. If a user asks a follow-up like "What about the 46mm version?", retrieval may miss context. Query rewriting (stretch goal) would fix this.
- `text-embedding-3-small` costs a small amount of money each time the vector store is rebuilt. The persistence check prevents unnecessary rebuilds.
